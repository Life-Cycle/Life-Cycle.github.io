nginx学习笔记

nginx
发布日期:   2020-09-19

正向代理反向代理

正向代理中服务端不知道客户端是谁，反向代理中客户端不知道服务端是谁。

正向代理

假如用户想访问Google，但是无法直接访问，这时候需要配置一个代理服务器，由代理服务器去访问Google，然后把结果返回给用户。在这个过程中，Google只知道是代理服务器访问了它，不知道具体的客户端是谁，这个代理过程就称为正向代理。

反向代理

我们去找中介租房，我们只访问了中介，不知道背后的房东是谁，中介就是反向代理。在这个过程中，我们不知道是谁给我们提供了房子，只能访问到中介，中介对我们隐藏了背后的资源。

反向代理常用在负载均衡中，比如eureka，nacos等框架就基于ribbon实现了负载均衡，ribbon就是反向代理。同一个微服务可能运行在服务器集群上的多台机器上，这时只需要将微服务以相同的微服务名注册到eureka或者nacos中，用户在访问该服务时，eureka或nacos就能实现负载均衡，轮询或随机或按设置的负载均衡算法让某一个微服务提供服务。

nginx配置

nginx配置在nginx.conf文件中，主要又三部分组成：

  • 全局块：从配置文件开始到event之间的内容，主要会设置一些影响到nginx服务器整体运行的配置指令。比如配置用户组，允许生成的worker process数，进程PID存放路径，日志存放路径和类型以及配置文件引入等。
  • events块：主要涉及到nginx服务器与用户的网络连接，这部分配置对nginx的性能影响较大，实际应用中要灵活配置。常用的设置包括是否开启对多work process下网络连接进行序列化，是否允许同时接收多个网络连接，选取哪种事件驱动模型来处理连接请求，每个word process可以同时支持的最大连接数等。
  • http块：这是nginx中配置最频繁的部分，代理，缓存日志定义等绝大多数功能和第三方模块的配置都在这里。
    需要注意的是http块也可以包括http全局块，server块。
    • http全局块：配置文件引入，MIME-TYPE定义，日志自定义，连接超时时间，单链接请求数上限等。
    • server块：从用户的角度看，每个server块就相当于一个硬件主机，每个http块可以包含多个server块，而每个server块又可以分为全局server块和location块。该技术产生是为了节省互联网服务器硬件成本。
      • 全局server块：一般用于配置监听端口，主机名称或ip配置
      • location块：一个server块可以有多个location块，这块的主要作用是接收nginx服务器接收到的请求字符串，比如server_name/uri-string，对主机名称(或IP地址)之外的字符串/uri-string进行匹配，对特定请求进行处理。地址定向，数据缓存和应答控制等功能，很多第三方模块的配置也在这里进行。
root@a3638f2e8d19:/etc/nginx# cat nginx.conf

user  nginx;
worker_processes  1; # 服务器并发处理的关键配置。值越大，表示可以支持的并发处理量越多

error_log  /var/log/nginx/error.log warn;
pid        /var/run/nginx.pid;


events {
    worker_connections  1024; # 表示每个worker process支持的最大连接数为1024

location指令说明

该指令用于匹配URL，语法如下：

location [=, ~ , ~* , ^~] /edu/ {
    ...
}
  1. =: 用于不含正则表达式的uri前，要求请求字符串与uri严格匹配，如果匹配成功，就停止继续向下搜索并立即处理该请求
  2. ~: 用于表示uri包含正则表达式，并且区分大小写
  3. ~*: 用于表示uri包含正则表达式，并且不区分大小写
  4. ^~: 用于不含正则表达式的uri前，要求Nginx服务器找到标识uri和请求字符串匹配度最高的location后，立即使用此location处理请求，而不再使用location块中的正则uri和请求字符串做匹配

注意: 如果uri包含正则表达式，则必须要有或者*标识。

负载均衡

配置负载均衡

配置负载均衡实现访问http://www.123.com/edu/a.html能够分别访问到8080和8081的tomcat服务。

server {
    listen       80;
    listen  [::]:80;
    server_name  localhost;

    # 1.配置upstream，取别名为myserver
    upstream myserver {
        server 172.17.0.5:8080;
        server 172.17.0.5:8081;
    }
    # 2.配置转发请求到myserver
    location / {
        proxy_pass http://myserver;
    }
}

负载均衡策略

很多专业的负载均衡硬件提供了很好的功能，但是价格不菲，这使得负载均衡软件大受欢迎。在linux下有Nginx、LVS、 Haproxy 等等服务可以提供负载均衡服务，而且Nginx提供了几种分配方式(策略):

  1. 轮询(默认)：每个请求按时间顺序逐一分配到不 同的后端服务器，如果后端服务器down掉，能自动剔除。

  2. weight(权重)：weight代表权重默认为1,权重越高被分配的客户端越多。
    指定轮询几率，weight和访问比率成正比，用于后端服务器性能不均的情况。例如: 

    upstream myserver {
    server 172.17.0.5:8080 weight=5;
    server 172.17.0.5:8081 weight=10;
}

  3. ip hash：每个请求按访问ip的hash结果分配, 这样每个访客固定访问一个后端服务器,可以解诀session的问题。例如:

    upstream server pool{
  ip_ hash
  server 192.168.5.21:80
  server 192.168.5.22:80
}

  4. fair (第三方)：按后端服务器的响应时间来分配请求，响应时间短的优先分配

    upstream server_pool 
    server 192.168.5.21:80;
    server 192.168.5.22:80;
    fair;
}

配置动静分离

动静分离简单来说就是把动态跟静态请求分开。不能理解成只是单独把动态页面和静态页面物理分离。严格意义上来说应该是把动态请求跟静态请求分开，可以理解成使用nginx处理静态页面，tomcat处理动态页面。动静分离从目前实现角度来讲大致分为两种：

一种是纯粹把静态文件独立成单独的域名，放在独立服务器上，也是目前主流推崇的方案；
另一种是吧动态跟静态文件混合在一起发布，通过nginx来分开。

通过location指定不同的后缀名实现不同的请求转发。通过expires参数设置，可以使浏览器缓存过期时间，减少与服务器之前的请求和流量。具体Expires定义: 是给一个资源设定一个过期时间，也就是说无需去服务端验证，直接通过浏览器自身确认是否过期即可，所以不会产生额外的流量。此种方法非常适合不经常变动的资源。(如果经常更新的文件，不建议使用Expires来缓存)，如果设置3d, 表示在这3天之内访问这个URL, 发送一个请求，比对服务器该文件最后更新时间没有变化，则不会从服务器抓取，返回状态码304，如果有修改，则直接从服务器重新下载，返回状态码200。

实例

在tomcat中webapps目录下新建/data/image和/data/www两个文件夹，然后在image目录里面放静态图片，在www目录里面放动态网页。当访问图片的时候直接返回图片，访问网页的时候转到网页，实现动静分离。

# nginx中配置如下
location /www/ {
    root /data/;
    index index.html index.htm;
}
location /image/ {
    root /data/;
    autoindex on;
}
# 实际测试分别访问一下两个地址，其中访问图片，如果路径问/image/则会列出图片列表
http://192.168.1.112/www/index.html
http://192.168.1.112/image/1.jpg

nginx原理

一个master多个worker的好处

可以使用nginx -s reload热部署。

首先，对于每个worker进程来说，独立的进程，不需要加锁，所以省掉了锁带来的开销，同时在编程以及问题查找时，也会方便很多。其次,采用独立的进程，可以让互相之间不会影响，一个进程退出后，其它进程还在工作，服务不会中断，master进程则很快启动新的worker进程。当然，worker进程的异常退出，肯定是程序有bug了，异常退出，会导致当前worker.上的所有请求失败，不过不会影响到所有请求，所以降低了风险。

设置多少个worker合适

Nginx同redis类似都采用了io多路复用机制，每个worker都是一个独立的进程， 但每个进程里只有一个主线程，通过异步非阻塞的方式来处理请求，即使是成千上万个请求也不在话下。每个worker的线程可以把一个cpu的性能发挥到极致。所以worker数和服务器的cpu数相等是最为适宜的。设少了会浪费cpu,设多了会造成cpu频繁切换上下文带来的损耗。

# 设置worker数量
worker.processes 4 

# work绑定cpu(4work绑定4cpu)
worker_cpu_affinity 0001 0010 0100 1000

# work绑定cpu (4work绑定8cpu中的4个)
worker_cpu_affinity 0000001 00000010 00000100 00001000

worker连接数

work_connections，这个值表示每个worker进程所能建立连接的最大值，所以，一个nginx 能建立的最大连接数，应该是worker.connections * worker processes。当然，这里说的是最大连接数，对于HTTP 请求本地资源来说，能够支持的最大并发数量是worker.connections * worker processes,如果是支持http1.1的浏览器每次访问要占两个连接，所以普通的静态访问最大并发数是: worker.connections * worker.processes / 2, 而如果是HTTP作为反向代理来说，最大并发数量应该是worker.connections * worker_proceses/4. 因为作为反向代理服务器，每个并发会建立与客户端的连接和与后端服务的连接，会占用两个连接.

几个常见问题：

第一个: 发送请求，占用了woker的几个连接数?
答案: 2或者4个。

第二个: nginx有一个master,有四个woker,每个woker支持最大的连接数1024,支持的最大并发数是多少?
答案：普通的静态访问最大并发数是: worker connections * worker processes /2，
而如果是HTTP作为反向代理来说，最大并发数量应该是worker connections * worker processes/4
